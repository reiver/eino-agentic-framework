Eino Agentic Framework

agentic framework for the Go programming language

by Charles Iliya Krempeaux

Eino is an open-source, production-grade agentic, LLM, and AI application framework.

Eino is a framework for creating agentic harnesses. And includes things such as tools, and all the other features you would expect from an agentic framework.

You can more or less think of Eino as LangChain and LangGraph for the Go programming language (golang). Eino provides similar low-level primitives as LangChain (ChatModel, Tool, Retriever, Embedding), but provides type safety. Also, Eino shares with LangGraph a graph-based, stateful approach to cyclic workflows, but does it within a smaller and more efficient artificat.

Pronunciation

Eino is pronounced “I know” even though it is spelled in a way where you would expect it to be pronounced “A know”.

Creator

Eino was originally developed by people at ByteDance, and released under their CloudWeGo ecosystem.

Since then, many other people have contributed to its development.

Model Support

Some models that Eino supports is: Claude, DeepSeek Gemini, GLM (by using the Eino OpenAI component), Ollama, OpenAI, Qwen, and others.

Components

Eino has a number of useful building blocks:

  • chat models
  • embedding models
  • retrievers
  • document loaders
  • prompt templates
  • tools
  • memory
  • output parsers

In Eino, these are defined a Go interfaces. Which means you can easli swap implementations without changing your logic.

Orchestration

Eino supports multiple orchestration styles.

Linear (Chain) Orchestration

Eino supports the chain / linear style of orchestration.

This type of (chain / linear) orchestration (in Eino) can be good for:

  • Classification
  • RAG
  • Summarization

Directed Graph Orchestration

Eino also supports the directed graph style of orchestration.

This type of (directed graph) orchestration (in Eino) can be good for:

  • conditional routing
  • retries
  • loops
  • branching logic

adk.ChatModelAgent

The ChatModelAgent (struct) type is probably Eino's most important prebuilt component. Even in this trivial Eino program ChatModelAgent is wired to run the classic ReAct (Reason → Act → Observe) loop internally. If you later attach tools via ToolsConfig, the same agent decides when to call them and when to answer, with no extra orchestration code.

adk.Runner

The Runner (struct) type is what you actually invoke. It returns an iterator of events (rather than a single string). This iterator looping style of coding is common with Eino. It is the same interface whether the agent emits one message, a streams tokens, calls tools, or hands off to sub-agents. It makes it so this consumption iterator looping doesn't change much as your agent grows in complexity.
